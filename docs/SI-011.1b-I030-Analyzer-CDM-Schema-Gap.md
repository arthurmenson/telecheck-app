## Append-only + Tier 0/1/2 invariants

### Both entities

**Tier 0 — Fully append-only from INSERT** (every column immutable post-INSERT; no UPDATE path; no DELETE path):
- Both `forms_i030_exemption` and `forms_template_i030_exemption_binding` are evidence artifacts
- The exemption itself is the §13.2 governance review approval record; mutation breaks the evidence chain
- The binding is the at-publish-time record of which exemptions covered which findings; mutation breaks the publish-trail integrity

Re-issuing an exemption (e.g., after expiry, or for a different draft revision) creates a NEW row; never an UPDATE.

## Repository layer enforcement (pseudo-SQL)

```sql
-- forms_i030_exemption: BEFORE INSERT trigger forces DB-time + computes finding_fingerprint
-- defense-in-depth (caller-supplied fingerprint silently verified via DB function below).
CREATE OR REPLACE FUNCTION forms_i030_exemption_insert_guard()
RETURNS TRIGGER AS $$
DECLARE
  v_recomputed_fingerprint TEXT;
BEGIN
  NEW.requested_at := NOW();  -- DB-time anchor
  NEW.approved_at := NOW();
  NEW.created_at := NOW();
  -- Note: finding_fingerprint is computed by the analyzer (NOT by DB) because the
  -- offending subtree comes from the form-template JSON which the DB doesn't parse.
  -- DB cross-validation deferred to publish-time gate where analyzer reruns and
  -- the gate verifies exemption fingerprint = analyzer-emitted fingerprint.
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_i030_exemption_insert_lifecycle_guard
  BEFORE INSERT ON forms_i030_exemption
  FOR EACH ROW
  EXECUTE FUNCTION forms_i030_exemption_insert_guard();

-- forms_i030_exemption: full append-only enforcement
CREATE OR REPLACE FUNCTION forms_i030_exemption_reject_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'forms_i030_exemption(% / %): row is evidence artifact; no UPDATE/DELETE permitted (Tier 0 full append-only per I-003 audit_records precedent)', COALESCE(OLD.tenant_id, NEW.tenant_id), COALESCE(OLD.exemption_id, NEW.exemption_id);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_i030_exemption_append_only_update
  BEFORE UPDATE ON forms_i030_exemption
  FOR EACH ROW
  EXECUTE FUNCTION forms_i030_exemption_reject_mutation();

CREATE TRIGGER forms_i030_exemption_append_only_delete
  BEFORE DELETE ON forms_i030_exemption
  FOR EACH ROW
  EXECUTE FUNCTION forms_i030_exemption_reject_mutation();

-- forms_template_i030_exemption_binding: BEFORE INSERT trigger forces DB-time
-- + cross-validates that analyzer_finding_fingerprint matches the referenced exemption's
-- finding_fingerprint (defense-in-depth)
CREATE OR REPLACE FUNCTION forms_template_i030_exemption_binding_insert_guard()
RETURNS TRIGGER AS $$
DECLARE
  v_exemption_fingerprint TEXT;
  v_exemption_expires_at  TIMESTAMPTZ;
BEGIN
  NEW.created_at := NOW();
  -- Cross-validate binding's analyzer_finding_fingerprint vs exemption's finding_fingerprint
  SELECT finding_fingerprint, expires_at
    INTO v_exemption_fingerprint, v_exemption_expires_at
   FROM forms_i030_exemption
   WHERE tenant_id = NEW.tenant_id AND exemption_id = NEW.exemption_id;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'forms_template_i030_exemption_binding: exemption_id=% not found in tenant=%', NEW.exemption_id, NEW.tenant_id;
  END IF;
  IF NEW.analyzer_finding_fingerprint IS DISTINCT FROM v_exemption_fingerprint THEN
    RAISE EXCEPTION 'forms_template_i030_exemption_binding: analyzer_finding_fingerprint=% does not match exemption.finding_fingerprint=% (binding rejected; exemption is narrowly-scoped to one specific finding)', NEW.analyzer_finding_fingerprint, v_exemption_fingerprint;
  END IF;
  -- Stale-binding rejection: cannot bind an already-expired exemption
  IF v_exemption_expires_at <= NOW() THEN
    RAISE EXCEPTION 'forms_template_i030_exemption_binding: exemption_id=% has expired (expires_at=%); cannot bind expired exemption', NEW.exemption_id, v_exemption_expires_at;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_i030_exemption_binding_insert_lifecycle_guard
  BEFORE INSERT ON forms_template_i030_exemption_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_i030_exemption_binding_insert_guard();

-- forms_template_i030_exemption_binding: full append-only enforcement
CREATE OR REPLACE FUNCTION forms_template_i030_exemption_binding_reject_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'forms_template_i030_exemption_binding(% / %): row is evidence artifact; no UPDATE/DELETE permitted', COALESCE(OLD.tenant_id, NEW.tenant_id), COALESCE(OLD.binding_id, NEW.binding_id);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_i030_exemption_binding_append_only_update
  BEFORE UPDATE ON forms_template_i030_exemption_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_i030_exemption_binding_reject_mutation();

CREATE TRIGGER forms_template_i030_exemption_binding_append_only_delete
  BEFORE DELETE ON forms_template_i030_exemption_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_i030_exemption_binding_reject_mutation();
```

## FORMS_ENGINE v5.X I-030 detection-rules amendment

The 6 detection categories (per SI-011 source line 95-100) need canonical JSONPath detection rules. Proposed canonicalization for FORMS_ENGINE v5.X §I-030 amendment:

### Category 1: Branching

Detect: any `branching_logic` predicate referencing `patient.research_consent_status` or its dotted-path equivalents.

```
JSONPath patterns to match:
  $.branching_logic..*.condition[?(@.references_field == "patient.research_consent_status")]
  $.branching_logic..*.condition[?(@.references_field starts with "patient.research_consent_status.")]
```

### Category 2: Visibility

Detect: field-level `visible_if` referencing research consent.

```
JSONPath patterns to match:
  $.fields[*].visible_if[?(@.references_field == "patient.research_consent_status")]
  $.fields[*].visible_if[?(@.references_field starts with "patient.research_consent_status.")]
```

### Category 3: Validation

Detect: validation predicate referencing research consent.

```
JSONPath patterns to match:
  $.fields[*].validations[*].predicate[?(@.references_field == "patient.research_consent_status")]
  $.fields[*].validations[*].predicate[?(@.references_field starts with "patient.research_consent_status.")]
```

### Category 4: Eligibility / Triage

Detect: `eligibility_logic` predicate referencing research consent.

```
JSONPath patterns to match:
  $.eligibility_logic..*.predicate[?(@.references_field == "patient.research_consent_status")]
  $.eligibility_logic..*.predicate[?(@.references_field starts with "patient.research_consent_status.")]
```

### Category 5: Pricing / Commerce

Detect: `approval_governance.pricing_overrides` referencing research consent.

```
JSONPath patterns to match:
  $.approval_governance.pricing_overrides[*].condition[?(@.references_field == "patient.research_consent_status")]
```

### Category 6: Outcome Messaging

Detect: dynamic copy templates substituting research-consent values.

```
JSONPath patterns to match:
  $.presentation_content..*.template_substitutions[?(@.source_field == "patient.research_consent_status")]
  $.fields[*].outcome_messaging..*.template[?(@.references == "patient.research_consent_status")]
```

### Finding fingerprint computation

For each detected pattern, the analyzer emits a finding with:

```
finding = {
  category: <one of the 6 enum values>,
  jsonpath: <exact JSONPath where the violation appears in the source JSON>,
  offending_subtree_canonical_json: <canonical-JSON serialization of the offending subtree at that JSONPath>,
  finding_fingerprint: SHA-256(category || "|" || jsonpath || "|" || offending_subtree_canonical_json)
}
```

The fingerprint is the binding key between the analyzer finding and the `forms_i030_exemption.finding_fingerprint`. Any change to the offending subtree invalidates prior exemptions (because the fingerprint changes → exemption no longer matches → publish-time exact-set-match predicate rejects).

## Publish-path I-030 gate (informational; implementation lands at templateService.publishVersion)

The publish-time I-030 gate consumes the SI-011.1b row shapes + the analyzer:

1. **Run analyzer** on this draft revision; collect set of findings F = {f_1, f_2, ...}
2. **Load bound exemptions** E = `SELECT * FROM forms_template_i030_exemption_binding b JOIN forms_i030_exemption e ON (...) WHERE b.tenant_id = ctx.tenantId AND b.template_id = draft.template_id AND b.draft_revision_id = draft.revision_id AND e.expires_at > NOW()`
3. **Exact-set-match predicate**: for every f_i in F, require exists e_j in E where e_j.finding_fingerprint = f_i.finding_fingerprint
4. **Stale-exemption rejection**: any e_j in E with NO matching f_i in F (i.e., exemption bound to a finding the analyzer no longer emits) → reject publish with `forms.publish.i030_exemption_stale` audit + stop processing
5. **Unmatched finding rejection**: any f_i in F with NO matching e_j in E → reject publish with `forms.publish.i030_violation` audit including (category, jsonpath, finding_fingerprint)
6. **Approve emission**: on successful gate pass, emit `forms.publish.i030_exemption_consumed` audit for each (e_j, f_i) pair

Audit events deferred to a future AUDIT_EVENTS amendment SI (canonical IDs listed but not landed at SI-011.1b).
