## What this is

SC7 P-024 Sub-decision #5 ratified the L4 Mode 2 input contract conformance gate's substantive design. SI-011.1d closes the corresponding **canonical row shape gap**: the publish-time binding between a form template revision and the specific `ai_workflow_handler_registry` (handler_id, handler_version, signature_hash) it integrates with.

This SI mirrors SI-011.1c's binding-row pattern but for AI workflow handlers instead of marketing copy. The binding is immutable provenance + runtime drift detection anchor per SC7 Sub-decision #5 step 3.

FOREIGN KEY (handler_id, handler_version, handler_signature_hash) REFERENCES ai_workflow_handler_registry (handler_id, version, signature_hash)

  -- This 3-column composite FK REQUIRES a matching UNIQUE constraint on ai_workflow_handler_registry:
  --
  --   ALTER TABLE ai_workflow_handler_registry
  --     ADD CONSTRAINT ai_workflow_handler_registry_fk_target_unique

  -- This is a MUST-level requirement (mirror SI-011.1c R2 closure pattern); SI-016's

## Append-only invariants

### forms_template_mode_2_binding

**Tier 0 — Fully append-only from INSERT** (every column immutable post-INSERT; no UPDATE path; no DELETE path):
- The row IS the publish-time provenance evidence for Mode 2 integration
- Re-publishing a revision with a different handler binding produces a NEW binding row; old rows preserved as historical evidence
- Mirrors I-003 audit_records discipline + SI-015/SI-011.1c precedent

## Repository layer enforcement (pseudo-SQL)

```sql
-- BEFORE INSERT trigger: force evidence-grade timestamps + cross-validate the
-- referenced handler version + status + signature_hash + schema-subset compatibility.
CREATE OR REPLACE FUNCTION forms_template_mode_2_binding_insert_guard()
RETURNS TRIGGER AS $$
DECLARE
  v_handler_status                     TEXT;
  v_handler_actual_signature_hash      TEXT;
  v_handler_input_validator_schema     JSONB;
BEGIN
  NEW.published_at := NOW();
  NEW.created_at := NOW();
  -- Cross-validate the bound handler row: must exist with declared version + status='active' + signature_hash match
  SELECT status, signature_hash, input_validator_schema
    INTO v_handler_status, v_handler_actual_signature_hash, v_handler_input_validator_schema
   FROM ai_workflow_handler_registry
   WHERE handler_id = NEW.handler_id
     AND version = NEW.handler_version;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'forms_template_mode_2_binding: handler (id=%, version=%) not found in registry (binding rejected)', NEW.handler_id, NEW.handler_version;
  END IF;
  -- Validate status='active' at publish time (deprecated/retired handlers cannot bind)
  IF v_handler_status IS DISTINCT FROM 'active' THEN
    RAISE EXCEPTION 'forms_template_mode_2_binding: handler (id=%, version=%) status=% (must be active; binding rejected)', NEW.handler_id, NEW.handler_version, v_handler_status;
  END IF;
  -- Validate signature_hash match (caller-supplied must equal actual; defense-in-depth)
  IF NEW.handler_signature_hash IS DISTINCT FROM v_handler_actual_signature_hash THEN
    RAISE EXCEPTION 'forms_template_mode_2_binding: caller-supplied handler_signature_hash=% does not match actual ai_workflow_handler_registry.signature_hash=% for (handler_id=%, version=%) (binding rejected; signature_hash binding is the trust anchor for runtime drift detection)', NEW.handler_signature_hash, v_handler_actual_signature_hash, NEW.handler_id, NEW.handler_version;
  END IF;
  -- Validate schema-subset compatibility (R1 HIGH closure 2026-05-18): MUST-level
  -- at v0.1 (no v0.2 TODO). The publish-time binding row is the trust anchor for
  -- Mode 2 contract conformance; if subset validation is a stub, malformed/
  -- incompatible templates bind to active handlers and only fail later at dispatch
  -- (or worse, fail silently when handler ignores/misinterprets fields).
  IF NEW.template_input_schema IS NULL THEN
    RAISE EXCEPTION 'forms_template_mode_2_binding: template_input_schema must be non-NULL JSONB';
  END IF;
  -- The validate_json_schema_subset function is REQUIRED to be implemented as
  -- part of SI-011.1d's canonical migration (NOT deferred). Concrete contract:
  --   validate_json_schema_subset(p_template_schema JSONB, p_handler_schema JSONB) RETURNS BOOLEAN
  -- Returns TRUE iff p_template_schema is a structural subset of p_handler_schema per
  -- the unified subset semantics below (R1 MEDIUM closure 2026-05-18). Returns FALSE
  -- (with descriptive error via RAISE NOTICE) iff any subset rule is violated.
  PERFORM 1 FROM (SELECT validate_json_schema_subset(NEW.template_input_schema, v_handler_input_validator_schema) AS ok) v WHERE v.ok = TRUE;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'forms_template_mode_2_binding: template_input_schema is NOT a valid structural subset of handler''s input_validator_schema for (handler_id=%, version=%) (R1 HIGH closure: subset validation MUST-level; reject publish; see validate_json_schema_subset RAISE NOTICE for specific violation)', NEW.handler_id, NEW.handler_version;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_mode_2_binding_insert_lifecycle_guard
  BEFORE INSERT ON forms_template_mode_2_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_mode_2_binding_insert_guard();

-- Full append-only enforcement (UPDATE + DELETE rejected)
CREATE OR REPLACE FUNCTION forms_template_mode_2_binding_reject_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'forms_template_mode_2_binding(% / %): row is evidence artifact; no UPDATE/DELETE permitted (Tier 0 full append-only per I-003 audit_records precedent)', COALESCE(OLD.tenant_id, NEW.tenant_id), COALESCE(OLD.binding_id, NEW.binding_id);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_mode_2_binding_append_only_update
  BEFORE UPDATE ON forms_template_mode_2_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_mode_2_binding_reject_mutation();

CREATE TRIGGER forms_template_mode_2_binding_append_only_delete
  BEFORE DELETE ON forms_template_mode_2_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_mode_2_binding_reject_mutation();
```

## Publish-time L4 Mode 2 gate (informational; implementation lands at templateService.publishVersion)

Per SC7 Sub-decision #5 step 2's 5-step validation:

1. **(a) Schema well-formed**: `template_input_schema` is a syntactically valid JSON Schema (draft-2020-12). Service-layer JSON Schema validator.
2. **(b) Form-field cross-walk**: every `required` property in `template_input_schema` corresponds to a field actually collected in `presentation_content`. Field type compatibility verified. Service-layer cross-walk.
3. **(c) Handler resolves**: query `ai_workflow_handler_registry WHERE handler_id = $handler_id AND version = $handler_version AND status = 'active'`. Reject with `forms.publish.mode_2_contract_invalid` audit (reason='handler_not_found' OR 'handler_not_active') if missing/non-active.
4. **(d) Signature compatibility**: capture `signature_hash` from registry at publish time + INSERT into `forms_template_mode_2_binding.handler_signature_hash`. The binding INSERT trigger re-validates equality (defense-in-depth). Reject with reason='signature_drift' if caller-supplied differs from actual.
5. **(e) Schema-handler compatibility**: `template_input_schema` is a structural subset of handler's registered `input_validator_schema`. **Unified subset semantics (R1 MEDIUM closure 2026-05-18 — previous draft had contradictory rules on extra fields)**:
   - **Rule 1 (required-property coverage)**: every property in handler's schema marked `required` MUST also be present + marked `required` in template's schema. Reject if template omits a required handler property.
   - **Rule 2 (type compatibility)**: for every property present in BOTH schemas, the template's type MUST be compatible with the handler's type (template's allowed values are a subset of handler's allowed values, e.g., handler accepts `string`, template provides `string` OR enum-string-subset).
   - **Rule 3 (no extra required fields)**: template MUST NOT declare `required` properties that don't exist in the handler's schema (forbidden — would either fail dispatch when handler rejects the unknown required field OR be silently dropped, creating hidden data-loss).
   - **Rule 4 (optional additional properties prohibited at v1.0)**: template MUST NOT declare additional properties (`required` or `optional`) that aren't in the handler's schema. Templates that need fields beyond the handler's contract MUST request a NEW handler version that includes those fields. This is stricter than the previous draft's "handler will ignore extras" — silent ignoring creates hidden data-loss bugs; explicit rejection forces the operator to register a coherent handler version.
   - Reject publish with `forms.publish.mode_2_contract_invalid` audit + `reason='schema_subset_violation'` + which-rule-failed detail.

On successful resolution → INSERT binding row. Failed validation → reject publish + audit.

**Runtime drift detection** (separate code path; R1 MEDIUM-2 + R2 MEDIUM closure 2026-05-18):
- At Mode 2 dispatch time, look up the binding row by **tenant-scoped** key `(tenant_id, template_id, publish_revision_id)` — `tenant_id` MUST be supplied from the dispatch context (`ctx.tenantId`). No fallback lookup by template/revision alone (would let an implementation path resolve another tenant's binding or report false drift against the wrong handler since `template_id` + `publish_revision_id` are not globally unique).
- Compare `binding.handler_signature_hash` to CURRENT `ai_workflow_handler_registry.signature_hash` for (handler_id, version)
- **Runtime deprecated-handler policy (R2 MEDIUM unified closure 2026-05-18):** dispatchable until `retired` per SC7 P-024 Sub-decision #5 step 4 + SI-016 OQ5 ratified default + SI-011 source line 148:
  - Handler status `active`: dispatch proceeds normally
  - Handler status `deprecated`: dispatch proceeds + emits `ai_workflow.handler_deprecated_dispatch_warning` Cat B audit per dispatch (operational continuity preserved; warning surface for migration planning); dispatch is NOT rejected
  - Handler status `retired`: reject dispatch + emit `ai_workflow.contract_drift_detected` audit (per I-012 reject-unless violation handling); audit carries `(tenant_id, template_id, publish_revision_id, handler_id, handler_version, expected_signature_hash, observed_signature_hash, observed_status='retired', drift_reason='handler_retired')` payload
  - Handler `signature_hash` mismatch (any status): reject dispatch + same audit with `drift_reason='signature_drift'`
- Previous draft + OQ3 had contradictory guidance; this unified policy resolves the contradiction at v0.1 ratification level: deprecated = dispatchable+warned; retired = rejected.
