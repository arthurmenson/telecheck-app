## Proposed canonical row shapes (FOR REVIEW — pre-Codex pre-ratification gate)

### §4.25 SyncSessionParticipants (entity #25)

```sql
-- v0.1 placeholder columns; SI-009.1 pre-ratification gate
id                          VARCHAR(26)  PRIMARY KEY  -- 'ssp_<ULID>'
tenant_id                   TEXT         NOT NULL REFERENCES tenants(id)
sync_session_id             VARCHAR(26)  NOT NULL  -- composite FK below
account_id                  VARCHAR(26)  NOT NULL  -- composite FK to accounts(tenant_id, account_id)
role                        VARCHAR(20)  NOT NULL CHECK (role IN ('patient', 'caregiver', 'clinician', 'interpreter', 'observer'))
joined_at                   TIMESTAMPTZ  NULL  -- when participant joined the LiveKit room
left_at                     TIMESTAMPTZ  NULL  -- when participant disconnected
created_at                  TIMESTAMPTZ  NOT NULL DEFAULT NOW()  -- scheduled-participant record creation
removed_at                  TIMESTAMPTZ  NULL  -- participant removed from scheduled participant list (vs left voluntarily)
removed_reason              VARCHAR(40)  NULL CHECK (removed_reason IS NULL OR removed_reason IN ('patient_request', 'clinician_request', 'system_removal', 'scheduling_conflict'))

-- Cross-tenant safety constraints (NOT placeholders; permanent)
UNIQUE (tenant_id, id)  -- composite UNIQUE for cross-entity composite FK safety
UNIQUE (tenant_id, sync_session_id, id)  -- triple-composite UNIQUE for future cross-entity reference
UNIQUE (tenant_id, sync_session_id, account_id)  -- one participant row per account per session

FOREIGN KEY (tenant_id, sync_session_id) REFERENCES sync_sessions (tenant_id, id)
FOREIGN KEY (tenant_id, account_id) REFERENCES accounts (tenant_id, account_id)

-- Removed-reason consistency CHECK (per 6d):
CHECK ((removed_at IS NULL AND removed_reason IS NULL) OR (removed_at IS NOT NULL AND removed_reason IS NOT NULL))

-- Lifecycle monotonicity CHECKs (Codex-Gate R1 MEDIUM-1 closure 2026-05-18):
-- Previously the row-shape declined DB CHECK enforcement for the
-- (joined_at, left_at, removed_at) tuple. Codex correctly identified that
-- this leaves direct-SQL / migration / admin-tool paths able to create
-- structurally-valid-but-semantically-impossible rows (e.g. left_at
-- populated without joined_at; left_at < joined_at; etc.).
-- These CHECKs enforce monotonicity at the DB layer:
CHECK (left_at IS NULL OR joined_at IS NOT NULL)
CHECK (joined_at IS NULL OR left_at IS NULL OR left_at >= joined_at)
CHECK (removed_at IS NULL OR removed_at >= created_at)
```

### §4.26 SyncSessionRecordings (entity #26)

```sql
-- v0.1 placeholder columns; SI-009.1 pre-ratification gate
id                              VARCHAR(26)  PRIMARY KEY  -- 'ssr_<ULID>'
tenant_id                       TEXT         NOT NULL REFERENCES tenants(id)
sync_session_id                 VARCHAR(26)  NOT NULL  -- composite FK below
s3_bucket                       TEXT         NOT NULL  -- per-tenant bucket per ADR-024
s3_key                          TEXT         NOT NULL  -- object key for the encrypted recording file (SSE-KMS at bucket level)
kms_key_id                      TEXT         NOT NULL  -- envelope traceability (per 7e — 2-column envelope)
kms_key_version                 INTEGER      NOT NULL  -- envelope traceability
recording_started_at            TIMESTAMPTZ  NOT NULL
recording_ended_at              TIMESTAMPTZ  NULL  -- nullable until recording finalized
byte_size                       BIGINT       NULL  -- post-finalization
retention_until                 TIMESTAMPTZ  NOT NULL  -- computed at recording-start from CCR retention key (per 7d — immutable)
retention_policy_version        INTEGER      NOT NULL  -- CCR snapshot version at recording-start (per 7d — immutable)
state                           VARCHAR(20)  NOT NULL CHECK (state IN ('recording', 'finalized', 'retention_expired', 'purged'))
patient_consent_audit_id        VARCHAR(26)  NOT NULL  -- FK to audit row capturing patient consent at session start (per 7c)
created_at                      TIMESTAMPTZ  NOT NULL DEFAULT NOW()
updated_at                      TIMESTAMPTZ  NOT NULL DEFAULT NOW()

-- Cross-tenant safety constraints (NOT placeholders; permanent)
UNIQUE (tenant_id, id)
UNIQUE (tenant_id, sync_session_id, id)  -- per 7a 1:N: multiple recordings per session allowed; this is for uniqueness of the recording row within a session

FOREIGN KEY (tenant_id, sync_session_id) REFERENCES sync_sessions (tenant_id, id)
FOREIGN KEY (tenant_id, patient_consent_audit_id) REFERENCES audit_records (tenant_id, audit_id)

-- State-machine consistency CHECK (R1 Codex closure 2026-05-17 —
-- tightened terminal-state branches to require finalized-object
-- metadata: a recording that never finalized cannot legitimately reach
-- retention_expired or purged. Without this, a failed-upload row could
-- be marked retention_expired with byte_size IS NULL — structurally
-- blessing incomplete uploads as terminal + making audit/compliance
-- recovery ambiguous):
CHECK (
  (state = 'recording'
    AND recording_ended_at IS NULL
    AND byte_size IS NULL)
  OR (state = 'finalized'
    AND recording_ended_at IS NOT NULL
    AND byte_size IS NOT NULL)
  OR (state = 'retention_expired'
    AND recording_ended_at IS NOT NULL
    AND byte_size IS NOT NULL          -- finalized-object metadata preserved
    AND retention_until <= NOW())
  OR (state = 'purged'
    AND recording_ended_at IS NOT NULL
    AND byte_size IS NOT NULL)         -- finalized-object metadata preserved
)

-- Monotonic-lifecycle enforcement (repository layer):
-- The state-machine consistency CHECK above allows rows to be INSERTed
-- directly into any state where the column-value preconditions match
-- (a row could theoretically be INSERTed at state='finalized' if
-- byte_size is populated at insert time). This is acceptable for the
-- DB layer because the CHECK still enforces internal consistency. The
-- repository layer enforces monotonic state transitions via the
-- standard UPDATE ... WHERE state IN (<allowed_prior_set>) guard
-- pattern (mirrors SI-007 + SI-008 append-only enforcement):
--   recording  → finalized          (via finalize_recording())
--   finalized  → retention_expired  (via retention background job)
--   retention_expired → purged      (via purge background job)
--   recording  → (no other transitions allowed; failed uploads use a
--                 separate sync_session_recording_failed_upload_log
--                 operational table to avoid polluting the canonical
--                 lifecycle with non-finalized entries)
-- An INSERT-direct-to-terminal-state path is rejected at the service
-- layer; only the background jobs operating on already-finalized rows
-- can advance state to retention_expired or purged.
```

### Append-only invariants (R2 Codex closure 2026-05-17 — narrowed to immutable payload/metadata fields to avoid blocking required compliance transitions)

The R1 wording asserted blanket-append-only on `sync_session_recordings` at `state IN ('finalized', 'retention_expired', 'purged')` — but the retention background job MUST transition `finalized → retention_expired` and the purge background job MUST transition `retention_expired → purged`. Blanket append-only would block both required compliance transitions. The R2 narrowing distinguishes between **payload/metadata immutability** (everything that defines the recording itself) and **state-machine progression** (which is explicitly allowed via guarded background jobs).

**`sync_session_participants`** (R3 + R4 Codex closures 2026-05-17 — narrowed to a three-stage one-shot lifecycle to accommodate the full LiveKit-room lifecycle: scheduled → joined → left → (optionally) removed; previous R3 wording froze `left_at` immediately after `joined_at` was set, blocking the canonical join→leave transition):

**Tier 0 — Identity binding immutable from INSERT** (these columns are FROZEN at row creation; never mutable):
- `sync_session_id`, `account_id`, `role` — participant identity binding cannot be retroactively changed
- `created_at` — record-creation timestamp is immutable

**Tier 1 — Three one-shot lifecycle transitions** (each is a NULL → NOT NULL transition guarded by the appropriate prior-state predicate; once each transition fires, that column is frozen):

| Transition | Allowed UPDATE guard | Effect |
| --- | --- | --- |
| **Join** | `WHERE joined_at IS NULL AND removed_at IS NULL` | Sets `joined_at = NOW()`; participant entered LiveKit room. **R5 closure 2026-05-17:** `removed_at IS NULL` predicate added — a participant who was removed-before-attendance cannot subsequently join (post-removal Tier 2 append-only enforced). |
| **Leave** | `WHERE joined_at IS NOT NULL AND left_at IS NULL AND removed_at IS NULL` | Sets `left_at = NOW()`; participant disconnected (requires having joined first + not yet removed). **R5 closure:** `removed_at IS NULL` predicate added (defense in depth; removal during attendance is unusual but possible if clinician boots a mid-session participant). |
| **Remove-after-leave** | `WHERE left_at IS NOT NULL AND removed_at IS NULL` | Sets `removed_at = NOW() + removed_reason`; formal removal from schedule after voluntary leave. (Already had `removed_at IS NULL` guard.) |
| **Remove-before-attendance** | `WHERE joined_at IS NULL AND removed_at IS NULL` | Sets `removed_at = NOW() + removed_reason`; participant removed from schedule before they ever joined. (Already had `removed_at IS NULL` guard.) |

Each transition is **one-shot**: once the target column is non-NULL, subsequent UPDATEs to that column are rejected by the guard. `removed_at` has two valid prior-states (NULL with no attendance, OR NULL after voluntary leave) — the row-shape contract documents both.

**Tier 2 — Fully append-only** (once `removed_at IS NOT NULL` OR `left_at IS NOT NULL AND removed_at IS NULL` AND session has ended, the row is terminal):
- After `removed_at IS NOT NULL`: fully append-only; cannot undo formal removal
- After `left_at IS NOT NULL` (without removal): the row is effectively terminal for attendance purposes; only the optional remove-after-leave UPDATE may fire

**Repository layer enforcement** (pseudo-SQL):
```sql
-- Tier 0 identity immutability (rejected for ALL post-INSERT mutations):
UPDATE sync_session_participants
   SET role = $new_role  -- ANY Tier 0 column
 WHERE id = $id
   AND tenant_id = $tenant_id
   -- NO guard required: Tier 0 columns have no allowed-mutation predicate
RETURNING id;
-- Tier 0 is enforced via BEFORE UPDATE trigger that REJECTS any UPDATE
-- where NEW.role IS DISTINCT FROM OLD.role (or any Tier 0 column changes);
-- mirrors SI-008 supersedes_execution_id BEFORE UPDATE trigger pattern.

-- Tier 1 Join transition (one-shot; rejected if previously removed-before-attendance):
UPDATE sync_session_participants
   SET joined_at = NOW(),
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND joined_at IS NULL
   AND removed_at IS NULL  -- R5 closure: post-removal Tier 2 append-only enforced
RETURNING id;
-- Zero-row return if joined_at IS NOT NULL (already joined) OR removed_at IS NOT NULL (cannot join after removal)

-- Tier 1 Leave transition (one-shot; requires prior join; defense-in-depth removed_at IS NULL):
UPDATE sync_session_participants
   SET left_at = NOW(),
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND joined_at IS NOT NULL
   AND left_at IS NULL
   AND removed_at IS NULL  -- R5 closure: defense-in-depth (clinician mid-session boot)
RETURNING id;
-- Zero-row return if joined_at IS NULL (never joined) OR left_at IS NOT NULL (already left) OR removed_at IS NOT NULL

-- Tier 1 Remove-after-leave transition (one-shot; requires prior leave):
UPDATE sync_session_participants
   SET removed_at = NOW(),
       removed_reason = $reason,
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND left_at IS NOT NULL
   AND removed_at IS NULL
RETURNING id;

-- Tier 1 Remove-before-attendance transition (one-shot; participant removed before joining):
UPDATE sync_session_participants
   SET removed_at = NOW(),
       removed_reason = $reason,
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND joined_at IS NULL
   AND removed_at IS NULL
RETURNING id;
```

This three-stage one-shot model honors the full LiveKit-room lifecycle (scheduled → joined → left → optionally formally-removed) while preserving Tier 0 identity-binding immutability + each lifecycle column's one-shot transition guarantee. Mirrors the `sync_session_recordings` two-tier pattern (Tier 1 payload immutability after finalization; Tier 2 guarded state-machine progression) but with a finer-grained three-stage lifecycle because participants have more distinct lifecycle stages than recordings.

**Non-bypassable Tier 2 enforcement** (R5 closure 2026-05-17 — DB-layer BEFORE UPDATE trigger as defense in depth against repository guard bugs OR direct-SQL/migration paths bypassing the repository layer):

```sql
CREATE OR REPLACE FUNCTION sync_session_participants_reject_post_removal_update()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.removed_at IS NOT NULL THEN
    -- Tier 2 fully append-only post-removal; reject ALL UPDATEs except
    -- updated_at-only no-op refreshes (which are themselves rejected
    -- since updated_at without a meaningful column change is pointless).
    RAISE EXCEPTION 'sync_session_participants(% / %): row is post-removal (removed_at = %); no UPDATEs permitted (Tier 2 append-only enforced at DB layer per SI-009.1 R5 closure)', NEW.tenant_id, NEW.id, OLD.removed_at;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_session_participants_tier2_append_only
  BEFORE UPDATE ON sync_session_participants
  FOR EACH ROW
  EXECUTE FUNCTION sync_session_participants_reject_post_removal_update();
```

Combined with the Tier 0 BEFORE UPDATE trigger (rejects any change to sync_session_id / account_id / role / created_at), the participant lifecycle invariants are enforced at TWO layers: (a) repository-layer guards prevent the canonical lifecycle method paths from violating the rules; (b) DB-layer BEFORE UPDATE triggers reject any UPDATE that bypasses the repository (direct SQL / migration / admin tool / future-developer-misunderstanding). This is the same defense-in-depth pattern SI-008 uses for `record_workflow_pointer_swap()` (declarative FK + procedure + trigger).

**Additional DB-layer triggers (Codex-Gate R1 MEDIUM-1 closure 2026-05-18):** the row-shape CHECKs above prevent semantically-impossible NEW rows, but two additional guards are needed to fully close direct-SQL bypass:

```sql
-- BEFORE INSERT trigger: reject pre-seeded lifecycle columns
-- (joined_at / left_at / removed_at MUST all be NULL at INSERT; lifecycle
-- columns become non-NULL ONLY via the guarded Tier 1 transition methods).
-- Without this, a direct INSERT could create a row with left_at already
-- populated (pretending the participant joined and left without auditable
-- transition events).
CREATE OR REPLACE FUNCTION sync_session_participants_reject_pre_seeded_lifecycle()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.joined_at IS NOT NULL OR NEW.left_at IS NOT NULL OR NEW.removed_at IS NOT NULL THEN
    RAISE EXCEPTION 'sync_session_participants(% / %): direct INSERT cannot pre-seed lifecycle columns (joined_at=% / left_at=% / removed_at=%); lifecycle columns become NON-NULL only via the guarded Tier 1 transition methods (Join, Leave, Remove-after-leave, Remove-before-attendance) — MEDIUM-1 closure', NEW.tenant_id, NEW.id, NEW.joined_at, NEW.left_at, NEW.removed_at;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_session_participants_insert_lifecycle_guard
  BEFORE INSERT ON sync_session_participants
  FOR EACH ROW
  EXECUTE FUNCTION sync_session_participants_reject_pre_seeded_lifecycle();

-- BEFORE UPDATE trigger: enforce one-shot direction for lifecycle columns
-- (NULL → NOT NULL is permitted via the Tier 1 transitions; NOT NULL → NULL
-- is forbidden — once set, lifecycle timestamps are append-only-direction).
-- This complements the existing post-removal Tier 2 trigger by closing the
-- earlier window: between INSERT and Tier 2 terminal state, lifecycle
-- columns must only progress, never reverse.
CREATE OR REPLACE FUNCTION sync_session_participants_reject_lifecycle_reversal()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.joined_at IS NOT NULL AND NEW.joined_at IS NULL THEN
    RAISE EXCEPTION 'sync_session_participants(% / %): joined_at cannot be reverted NOT NULL → NULL (one-shot Tier 1 invariant; MEDIUM-1 closure)', NEW.tenant_id, NEW.id;
  END IF;
  IF OLD.left_at IS NOT NULL AND NEW.left_at IS NULL THEN
    RAISE EXCEPTION 'sync_session_participants(% / %): left_at cannot be reverted NOT NULL → NULL (one-shot Tier 1 invariant; MEDIUM-1 closure)', NEW.tenant_id, NEW.id;
  END IF;
  IF OLD.removed_at IS NOT NULL AND NEW.removed_at IS NULL THEN
    RAISE EXCEPTION 'sync_session_participants(% / %): removed_at cannot be reverted NOT NULL → NULL (one-shot Tier 1 invariant; MEDIUM-1 closure)', NEW.tenant_id, NEW.id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_session_participants_update_lifecycle_direction_guard
  BEFORE UPDATE ON sync_session_participants
  FOR EACH ROW
  EXECUTE FUNCTION sync_session_participants_reject_lifecycle_reversal();
```

**Trigger interaction:** the existing Tier 2 post-removal trigger fires only when `OLD.removed_at IS NOT NULL`; the new one-shot-direction trigger fires for all UPDATEs and catches NULL-reversal attempts at any lifecycle stage. The INSERT trigger forces all new rows to start at the canonical scheduled-only state. Triple-layered enforcement: row CHECKs (monotonicity) + INSERT trigger (no pre-seeding) + UPDATE triggers (one-shot direction + Tier 0 immutability + Tier 2 post-removal).

**`sync_session_recordings`** (two-tier invariant):

**Tier 1 — Payload + metadata immutability** (once `state = 'finalized'` is reached, the following columns are FROZEN):
- `s3_bucket`, `s3_key` — recording location is immutable (any change implies a different recording)
- `kms_key_id`, `kms_key_version` — encryption envelope is immutable (key rotation creates a NEW recording row, never mutates the existing one)
- `byte_size` — finalized object size is immutable (the recording has been uploaded + its size is durable)
- `recording_started_at`, `recording_ended_at` — temporal anchors are immutable (timestamps cannot be retroactively changed)
- `retention_until`, `retention_policy_version` — retention is locked at recording-START per ratified sub-decision #7d; the retention window CANNOT be extended or shortened post-finalization (per 7d rationale: floating retention is a compliance hazard)
- `patient_consent_audit_id` — consent evidence is immutable (the consent audit row itself is append-only per I-027)

**Tier 2 — Allowed state-machine transitions** (state column + state-transition timestamps remain mutable, but ONLY through guarded background jobs):
- `finalized → retention_expired`: triggered by retention background job when `retention_until <= NOW()`. Allowed via `UPDATE ... WHERE state = 'finalized' AND retention_until <= NOW()` guard. Emits Cat A `sync_session_recording.retention_expired` audit.
- `retention_expired → purged`: triggered by purge background job after S3 object purge succeeds. Allowed via `UPDATE ... WHERE state = 'retention_expired'` guard. Emits Cat A `sync_session_recording.purged` audit with `(purged_at, purge_actor_id, purge_method)` detail.
- `updated_at`: bumped on each allowed state transition for operational tracking.
- All other columns (Tier 1 list above): FROZEN; any attempted UPDATE rejected at repository layer.

**Terminal-state finalization invariant** (R1 closure preserved):
- `state = 'retention_expired'` REQUIRES `byte_size IS NOT NULL` (recording must have finalized; failed-upload rows route to separate operational table)
- `state = 'purged'` REQUIRES `byte_size IS NOT NULL` (same)
- Direct INSERT-to-terminal-state rejected at service layer (only background jobs advancing from finalized → retention_expired → purged can produce terminal-state rows)

**Repository layer enforcement** (pseudo-SQL):
```sql
-- Tier 1 payload immutability (rejects ALL state-bearing transitions on frozen columns):
UPDATE sync_session_recordings
   SET s3_bucket = $new_s3_bucket  -- ANY frozen column
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND state = 'recording'  -- frozen ONLY after finalization
RETURNING id;
-- Zero-row return if state IN ('finalized', 'retention_expired', 'purged')

-- Tier 2 retention transition (allowed via background job):
UPDATE sync_session_recordings
   SET state = 'retention_expired',
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND state = 'finalized'
   AND retention_until <= NOW()
RETURNING id;

-- Tier 2 purge transition (allowed via background job after S3 purge succeeds):
UPDATE sync_session_recordings
   SET state = 'purged',
       updated_at = NOW()
 WHERE id = $id
   AND tenant_id = $tenant_id
   AND state = 'retention_expired'
RETURNING id;
```

This two-tier model preserves the append-only guarantee Codex's R1 finding was protecting (payload + retention envelope cannot be retroactively changed once finalized) while honoring the operational reality that retention/purge compliance REQUIRES state-machine progression after finalization.

### DB-layer state-machine + payload-immutability triggers for `sync_session_recordings` (Codex-Gate R1 HIGH-1 + HIGH-2 closures 2026-05-18)

Codex R1 HIGH-1 + HIGH-2 correctly identified that the repository-layer guarded UPDATE pattern is insufficient on its own:
- A direct SQL path, migration, admin tool, or compromised writer could INSERT a row with `state='purged'` + finalized-object metadata, marking the recording as purged before retention expiry (still passing the CHECK constraint above)
- The same paths could UPDATE finalized rows' Tier 1 frozen columns (s3 location, KMS envelope, retention_until, etc.) without triggering the repository's zero-row guard

Defense-in-depth at the DB layer (mirroring the `sync_session_participants` Tier 0/Tier 2 trigger pattern):

```sql
-- BEFORE INSERT trigger: enforce that all new recordings start at state='recording'
-- (no direct-to-terminal-state INSERT path; only background jobs operating on
-- already-finalized rows can advance state).
CREATE OR REPLACE FUNCTION sync_session_recordings_reject_direct_terminal_insert()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.state IS DISTINCT FROM 'recording' THEN
    RAISE EXCEPTION 'sync_session_recordings(% / %): direct INSERT to non-recording state (%) is forbidden; new rows MUST start at state=''recording'' and progress via the guarded background jobs (finalize_recording → retention_expired → purged)', NEW.tenant_id, NEW.id, NEW.state;
  END IF;
  IF NEW.recording_ended_at IS NOT NULL OR NEW.byte_size IS NOT NULL THEN
    RAISE EXCEPTION 'sync_session_recordings(% / %): direct INSERT cannot pre-seed recording_ended_at or byte_size; these become NON-NULL only via the finalize_recording transition', NEW.tenant_id, NEW.id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_session_recordings_state_machine_insert_guard
  BEFORE INSERT ON sync_session_recordings
  FOR EACH ROW
  EXECUTE FUNCTION sync_session_recordings_reject_direct_terminal_insert();

-- BEFORE UPDATE trigger: enforce OLD.state → NEW.state transition allow-list
-- + Tier 1 frozen-column immutability when OLD.state is post-finalization.
CREATE OR REPLACE FUNCTION sync_session_recordings_enforce_lifecycle()
RETURNS TRIGGER AS $$
BEGIN
  -- State-machine progression allow-list (transition arcs explicit):
  IF OLD.state = 'recording' THEN
    -- Allowed: stay at 'recording' (no-op refreshes) OR advance to 'finalized'
    IF NEW.state NOT IN ('recording', 'finalized') THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): forbidden transition recording → %; only recording → finalized allowed', NEW.tenant_id, NEW.id, NEW.state;
    END IF;
    -- If advancing to 'finalized', require the finalize-metadata preconditions
    IF NEW.state = 'finalized' AND (NEW.recording_ended_at IS NULL OR NEW.byte_size IS NULL) THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): cannot transition recording → finalized without recording_ended_at AND byte_size populated', NEW.tenant_id, NEW.id;
    END IF;
  ELSIF OLD.state = 'finalized' THEN
    -- Allowed: stay at 'finalized' (no-op refreshes) OR advance to 'retention_expired'
    -- (only when retention_until <= NOW() — predicate enforced at trigger time)
    IF NEW.state NOT IN ('finalized', 'retention_expired') THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): forbidden transition finalized → %; only finalized → retention_expired allowed', NEW.tenant_id, NEW.id, NEW.state;
    END IF;
    IF NEW.state = 'retention_expired' AND NEW.retention_until > NOW() THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): cannot transition finalized → retention_expired before retention_until (%)', NEW.tenant_id, NEW.id, NEW.retention_until;
    END IF;
    -- Tier 1 frozen-column immutability post-finalization (HIGH-2 closure):
    IF NEW.s3_bucket IS DISTINCT FROM OLD.s3_bucket
       OR NEW.s3_key IS DISTINCT FROM OLD.s3_key
       OR NEW.kms_key_id IS DISTINCT FROM OLD.kms_key_id
       OR NEW.kms_key_version IS DISTINCT FROM OLD.kms_key_version
       OR NEW.byte_size IS DISTINCT FROM OLD.byte_size
       OR NEW.recording_started_at IS DISTINCT FROM OLD.recording_started_at
       OR NEW.recording_ended_at IS DISTINCT FROM OLD.recording_ended_at
       OR NEW.retention_until IS DISTINCT FROM OLD.retention_until
       OR NEW.retention_policy_version IS DISTINCT FROM OLD.retention_policy_version
       OR NEW.patient_consent_audit_id IS DISTINCT FROM OLD.patient_consent_audit_id THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): Tier 1 frozen-column immutability violated (state=% requires payload/metadata frozen; HIGH-2 closure)', NEW.tenant_id, NEW.id, OLD.state;
    END IF;
  ELSIF OLD.state = 'retention_expired' THEN
    -- Allowed: stay at 'retention_expired' (no-op refreshes) OR advance to 'purged'
    IF NEW.state NOT IN ('retention_expired', 'purged') THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): forbidden transition retention_expired → %; only retention_expired → purged allowed', NEW.tenant_id, NEW.id, NEW.state;
    END IF;
    -- Tier 1 frozen-column immutability (HIGH-2 closure; identical check)
    IF NEW.s3_bucket IS DISTINCT FROM OLD.s3_bucket
       OR NEW.s3_key IS DISTINCT FROM OLD.s3_key
       OR NEW.kms_key_id IS DISTINCT FROM OLD.kms_key_id
       OR NEW.kms_key_version IS DISTINCT FROM OLD.kms_key_version
       OR NEW.byte_size IS DISTINCT FROM OLD.byte_size
       OR NEW.recording_started_at IS DISTINCT FROM OLD.recording_started_at
       OR NEW.recording_ended_at IS DISTINCT FROM OLD.recording_ended_at
       OR NEW.retention_until IS DISTINCT FROM OLD.retention_until
       OR NEW.retention_policy_version IS DISTINCT FROM OLD.retention_policy_version
       OR NEW.patient_consent_audit_id IS DISTINCT FROM OLD.patient_consent_audit_id THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): Tier 1 frozen-column immutability violated (state=% requires payload/metadata frozen; HIGH-2 closure)', NEW.tenant_id, NEW.id, OLD.state;
    END IF;
  ELSIF OLD.state = 'purged' THEN
    -- Terminal state: only state-stable no-op UPDATEs allowed; all Tier 1 columns
    -- still frozen for forensic-walk integrity.
    IF NEW.state IS DISTINCT FROM 'purged' THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): state=purged is terminal; no transitions allowed', NEW.tenant_id, NEW.id;
    END IF;
    -- Tier 1 frozen-column immutability (HIGH-2 closure; identical check)
    IF NEW.s3_bucket IS DISTINCT FROM OLD.s3_bucket
       OR NEW.s3_key IS DISTINCT FROM OLD.s3_key
       OR NEW.kms_key_id IS DISTINCT FROM OLD.kms_key_id
       OR NEW.kms_key_version IS DISTINCT FROM OLD.kms_key_version
       OR NEW.byte_size IS DISTINCT FROM OLD.byte_size
       OR NEW.recording_started_at IS DISTINCT FROM OLD.recording_started_at
       OR NEW.recording_ended_at IS DISTINCT FROM OLD.recording_ended_at
       OR NEW.retention_until IS DISTINCT FROM OLD.retention_until
       OR NEW.retention_policy_version IS DISTINCT FROM OLD.retention_policy_version
       OR NEW.patient_consent_audit_id IS DISTINCT FROM OLD.patient_consent_audit_id THEN
      RAISE EXCEPTION 'sync_session_recordings(% / %): Tier 1 frozen-column immutability violated (state=% requires payload/metadata frozen; HIGH-2 closure)', NEW.tenant_id, NEW.id, OLD.state;
    END IF;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER sync_session_recordings_lifecycle_guard
  BEFORE UPDATE ON sync_session_recordings
  FOR EACH ROW
  EXECUTE FUNCTION sync_session_recordings_enforce_lifecycle();
```

**Effect of the trigger pair:**
- INSERT trigger: blocks direct-to-terminal-state INSERTs (could not previously create a `purged` row pre-finalization)
- UPDATE trigger: validates exact OLD.state → NEW.state transitions (eliminates direct-SQL paths skipping monotonic progression), enforces retention-window timing (`retention_until <= NOW()` predicate at trigger time, not just at repository-layer), AND enforces Tier 1 frozen-column immutability for any UPDATE where OLD.state ∈ {finalized, retention_expired, purged}. Repository-layer guards continue to operate at the application layer; trigger is defense-in-depth.

## Cross-tenant safety constraints (NOT placeholders; permanent)

Mirror of SI-007 / SI-008 / SI-009 / SI-005 discipline:

1. `sync_session_participants UNIQUE (tenant_id, id)` + `(tenant_id, sync_session_id, id)` + `(tenant_id, sync_session_id, account_id)`
2. `sync_session_participants FK (tenant_id, sync_session_id) → sync_sessions(tenant_id, id)` — same-tenant + same-session lineage
3. `sync_session_participants FK (tenant_id, account_id) → accounts(tenant_id, account_id)` — same-tenant account binding
4. `sync_session_recordings UNIQUE (tenant_id, id)` + `(tenant_id, sync_session_id, id)` (per 7a 1:N)
5. `sync_session_recordings FK (tenant_id, sync_session_id) → sync_sessions(tenant_id, id)` — same-tenant + same-session lineage
6. `sync_session_recordings FK (tenant_id, patient_consent_audit_id) → audit_records(tenant_id, audit_id)` — same-tenant consent-evidence binding

**Cross-tenant attack vector closed:** without these composite FKs, an admin-tool bug or malicious service-level call could create participants/recordings rows in tenant A pointing at tenant B's sync_session OR account OR audit row. RLS would let tenant A users see only their own row, but background jobs / cross-tenant queries could chain through the FK to reach tenant B's data. Composite FKs make cross-tenant references structurally impossible (per ADR-023 + PROJECT_CONVENTIONS r5 §1.1 + I-023..I-027).

## Cross-tenant safety regression test acceptance criteria

Per the SI-007/SI-008/SI-009 precedent — all three entities require regression tests asserting:

1. **Participant cross-tenant prevention**: cannot create a participant in tenant A pointing at tenant B's sync_session (composite FK violation)
2. **Participant account cross-tenant prevention**: cannot create a participant in tenant A pointing at tenant B's account
3. **Recording cross-tenant prevention**: cannot create a recording in tenant A pointing at tenant B's sync_session OR audit_record
4. **Participant uniqueness**: cannot add the same account twice to the same sync_session (idempotent participant add)
5. **Recording 1:N**: multiple recordings per sync_session legitimate (technical-failure-reschedule scenario)
6. **Append-only invariants (R2 + R3 Codex closures 2026-05-17 — narrowed two-tier model for BOTH entities):**
   - **Recordings (R2 closure):** (a) cannot UPDATE Tier 1 payload/metadata columns on a `sync_session_recordings` row after `state = 'finalized'` (s3_bucket, s3_key, kms_key_id, kms_key_version, byte_size, recording_started_at, recording_ended_at, retention_until, retention_policy_version, patient_consent_audit_id — all frozen); (b) Tier 2 state-machine progression IS PERMITTED via guarded background jobs (finalized → retention_expired → purged) with explicit prior-state guards; (c) direct INSERT-to-terminal-state OR direct UPDATE bypassing the two-tier rules is rejected at service + repository layers; (d) test BOTH: (i) rejected mutation of frozen payload columns post-finalization AND (ii) allowed state-machine progression via guarded background jobs.
   - **Participants (R3 + R4 closures):** three-stage one-shot lifecycle scheduled → joined → left → (optionally) removed. (e) Tier 0 identity columns (sync_session_id, account_id, role, created_at) IMMUTABLE from INSERT via BEFORE UPDATE trigger (mirrors SI-008 supersedes_execution_id pattern); (f) Tier 1 Join transition allowed via `WHERE joined_at IS NULL` guard (one-shot); (g) Tier 1 Leave transition allowed via `WHERE joined_at IS NOT NULL AND left_at IS NULL` guard (one-shot; requires prior join); (h) Tier 1 Remove-after-leave transition allowed via `WHERE left_at IS NOT NULL AND removed_at IS NULL` guard; (i) Tier 1 Remove-before-attendance transition allowed via `WHERE joined_at IS NULL AND removed_at IS NULL` guard; (j) Tier 2 fully append-only once `removed_at IS NOT NULL`; (k) test FULL lifecycle: (i) allowed Join, (ii) allowed Leave after Join, (iii) allowed Remove-after-Leave, (iv) allowed Remove-before-attendance, (v) rejected Leave without prior Join, (vi) rejected double Join / double Leave / double Remove, (vii) rejected mutation of Tier 0 identity columns via BEFORE UPDATE trigger, (viii) rejected post-removal UPDATE of any column INCLUDING rejected Join-after-Remove-before-attendance attempt (the R5 closure case: previously the Join guard only checked `joined_at IS NULL`; a participant removed-before-attendance with NULL joined_at would have matched the guard); (ix) DB-layer BEFORE UPDATE trigger `sync_session_participants_reject_post_removal_update()` rejects ANY UPDATE on a row with `removed_at IS NOT NULL` (defense in depth against repository guard bugs / direct-SQL paths / migration paths).
   - This narrowing avoids the R2 + R3 Codex findings that the original blanket append-only rules would block required compliance transitions (retention/purge background jobs MUST advance recording state from finalized; remove-after-left workflow MUST advance participant `removed_at` from NULL to NOT NULL after attendance).
7. **Terminal-state finalization invariant (R1 Codex closure 2026-05-17)**: cannot INSERT OR UPDATE a `sync_session_recordings` row to `state='retention_expired'` OR `state='purged'` if `byte_size IS NULL` (i.e., the recording never finalized). State-machine consistency CHECK enforces at DB layer. Monotonic-lifecycle enforcement at repository layer prevents direct-to-terminal INSERTs; only the retention background job advances `finalized → retention_expired`, and only the purge background job advances `retention_expired → purged`. Failed-upload rows route to a separate `sync_session_recording_failed_upload_log` operational table (NOT in canonical CDM scope; operational only). This closes the Codex R1 finding that allowed a recording that never finalized to be marked retention_expired or purged — structurally blessing incomplete uploads as terminal and making audit/compliance recovery ambiguous.
