## What this is

SC7 P-024 Sub-decision #4 ratified the L4 MarketingCopy approval gate substantive design. SI-011.1c closes the corresponding **canonical row shape gap**: the publish-time binding between a form template revision and the specific MarketingCopy versions referenced in its `presentation_content`.

This SI adds a single new entity that records the (template, revision) → (marketing_copy_id, version, content_fingerprint) binding at publish time. The binding is immutable provenance + the anchor for runtime drift detection per SC7 Sub-decision #4 step 4.

## Proposed canonical row shapes (FOR REVIEW — pre-Codex pre-ratification gate)

### §4.36 forms_template_marketing_copy_binding (entity #52)

```sql
-- v0.1 placeholder columns; SI-011.1c pre-ratification gate
binding_id                          VARCHAR(26)  PRIMARY KEY  -- 'fmcb_<ULID>'
tenant_id                           TEXT         NOT NULL REFERENCES tenants(id)
template_id                         VARCHAR(26)  NOT NULL
publish_revision_id                 VARCHAR(26)  NOT NULL  -- the revision_id AT PUBLISH TIME (frozen — published revision is immutable per I-013)
marketing_copy_id                   VARCHAR(26)  NOT NULL  -- FK to marketing_copy.id (mkc_<ULID>)
marketing_copy_version              TEXT         NOT NULL  -- semver string of the bound version
marketing_copy_content_fingerprint  TEXT         NOT NULL  -- SHA-256 captured at publish time (DB-owned per SI-015 R1 HIGH-2)
reference_locator                   TEXT         NOT NULL  -- RFC 6901 JSON Pointer to the marketing_copy_ref site in presentation_content (per SI-011.1b R3/R4 dialect-independent contract)
published_at                        TIMESTAMPTZ  NOT NULL DEFAULT NOW()  -- DB-clock anchor; forced via INSERT trigger
created_at                          TIMESTAMPTZ  NOT NULL DEFAULT NOW()  -- DB-clock anchor; forced via INSERT trigger

-- Cross-tenant safety constraints (NOT placeholders; permanent)
UNIQUE (tenant_id, binding_id)
UNIQUE (tenant_id, template_id, publish_revision_id, reference_locator)
  -- At most one binding per (template, publish_revision, reference_locator) — each marketing_copy_ref site in a published revision has exactly one binding

FOREIGN KEY (tenant_id, template_id, publish_revision_id) REFERENCES forms_template_revisions (tenant_id, template_id, revision_id)

-- Triple-composite FK to marketing_copy (R1 HIGH closure equivalent pre-emptive: MUST include all version-aware identifiers since marketing_copy PK is (id, version) per SI-015 §4.27):
FOREIGN KEY (tenant_id, marketing_copy_id) REFERENCES marketing_copy (tenant_id, id)
  -- Note: tenant_id binding makes cross-tenant reference structurally impossible (per SI-015 cross-tenant safety pattern)
  -- The marketing_copy_version is denormalized but joined-on at publish/runtime; the (tenant_id, marketing_copy_id, marketing_copy_version) tuple MUST resolve to exactly one marketing_copy row when validation runs

-- Timestamp monotonicity
CHECK (published_at >= created_at)
```

## Append-only invariants

### forms_template_marketing_copy_binding

**Tier 0 — Fully append-only from INSERT** (every column immutable post-INSERT; no UPDATE path; no DELETE path):
- The row IS the publish-time provenance evidence
- Re-publishing a revision with different MarketingCopy bindings produces a NEW binding row per new (publish_revision, reference_locator); the old binding rows stay as historical evidence of what was published when
- Mirrors I-003 audit_records discipline + SI-015 MarketingCopyGovernanceEvidence pattern

## Repository layer enforcement (pseudo-SQL)

```sql
-- BEFORE INSERT trigger: force evidence-grade timestamps + cross-validate that the
-- referenced marketing_copy version exists, is approved, and the fingerprint matches.
CREATE OR REPLACE FUNCTION forms_template_marketing_copy_binding_insert_guard()
RETURNS TRIGGER AS $$
DECLARE
  v_marketing_copy_status                  TEXT;
  v_marketing_copy_actual_fingerprint      TEXT;
  v_marketing_copy_actual_version          TEXT;
BEGIN
  NEW.published_at := NOW();
  NEW.created_at := NOW();
  -- Cross-validate the bound marketing_copy row: must exist in same tenant with the
  -- declared version + status='approved' + fingerprint matching the value bound here
  SELECT status, content_fingerprint, version
    INTO v_marketing_copy_status, v_marketing_copy_actual_fingerprint, v_marketing_copy_actual_version
   FROM marketing_copy
   WHERE tenant_id = NEW.tenant_id
     AND id = NEW.marketing_copy_id
     AND version = NEW.marketing_copy_version;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'forms_template_marketing_copy_binding: marketing_copy (id=%, version=%) not found in tenant=% (binding rejected)', NEW.marketing_copy_id, NEW.marketing_copy_version, NEW.tenant_id;
  END IF;
  IF v_marketing_copy_status IS DISTINCT FROM 'approved' THEN
    RAISE EXCEPTION 'forms_template_marketing_copy_binding: marketing_copy (id=%, version=%) status=% (must be approved; binding rejected)', NEW.marketing_copy_id, NEW.marketing_copy_version, v_marketing_copy_status;
  END IF;
  IF NEW.marketing_copy_content_fingerprint IS DISTINCT FROM v_marketing_copy_actual_fingerprint THEN
    RAISE EXCEPTION 'forms_template_marketing_copy_binding: caller-supplied marketing_copy_content_fingerprint=% does not match actual marketing_copy.content_fingerprint=% for (id=%, version=%) (binding rejected; fingerprint binding is the trust anchor for runtime drift detection)', NEW.marketing_copy_content_fingerprint, v_marketing_copy_actual_fingerprint, NEW.marketing_copy_id, NEW.marketing_copy_version;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_marketing_copy_binding_insert_lifecycle_guard
  BEFORE INSERT ON forms_template_marketing_copy_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_marketing_copy_binding_insert_guard();

-- Full append-only enforcement (UPDATE + DELETE rejected)
CREATE OR REPLACE FUNCTION forms_template_marketing_copy_binding_reject_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'forms_template_marketing_copy_binding(% / %): row is evidence artifact; no UPDATE/DELETE permitted (Tier 0 full append-only per I-003 audit_records precedent)', COALESCE(OLD.tenant_id, NEW.tenant_id), COALESCE(OLD.binding_id, NEW.binding_id);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER forms_template_marketing_copy_binding_append_only_update
  BEFORE UPDATE ON forms_template_marketing_copy_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_marketing_copy_binding_reject_mutation();

CREATE TRIGGER forms_template_marketing_copy_binding_append_only_delete
  BEFORE DELETE ON forms_template_marketing_copy_binding
  FOR EACH ROW
  EXECUTE FUNCTION forms_template_marketing_copy_binding_reject_mutation();
```

## Publish-time L4 gate (informational; implementation lands at templateService.publishVersion)

1. **Extract marketing_copy_refs** from the draft revision's `presentation_content`. Each reference is an object `{"type": "marketing_copy_ref", "id": "mkc_<ULID>"}`. Use RFC 6901 JSON Pointer to record the reference site location (R3/R4 SI-011.1b dialect-independent contract; same canonicalization pattern).
2. **For each reference**, resolve the referenced marketing_copy:
   - Query: `SELECT id, version, content_fingerprint, status FROM marketing_copy WHERE tenant_id = ctx.tenantId AND id = ref.id ORDER BY version DESC LIMIT 1` (latest version in same tenant)
   - Validate `tenant_id = ctx.tenantId` (cross-tenant references categorically forbidden per SI-015 cross-tenant safety)
   - Validate `status = 'approved'`
   - Capture `content_fingerprint` value at THIS publish time
3. **Reject publish** if any reference fails resolution with `forms.publish.marketing_copy_not_approved` audit specifying the failing reference `id` + reason (`missing`, `not_approved`, `cross_tenant`, `fingerprint_drift` — for the publish-time fingerprint capture path the latter is impossible by construction since we read from marketing_copy directly, but defense-in-depth records the reason).
4. **On successful resolution**, INSERT a `forms_template_marketing_copy_binding` row for each reference. The BEFORE INSERT trigger re-validates the binding (defense-in-depth on top of step 2). After all bindings INSERT, the publish proceeds.
5. **Runtime rendering drift detection** (separate code path; not part of publish gate):
   - At render time, look up the binding row for the (template, publish_revision, reference_locator)
   - Compare `binding.marketing_copy_content_fingerprint` to the CURRENT `marketing_copy.content_fingerprint` for the same (tenant, id, version)
   - On mismatch: reject render + emit `forms.runtime.marketing_copy_drift_detected` audit specifying expected + observed fingerprints

## Cross-tenant safety constraints summary

| # | FK / UNIQUE | Source | Target | Same-tenant lineage |
| --- | --- | --- | --- | --- |
| 1 | FK | `forms_template_marketing_copy_binding.(tenant_id, template_id, publish_revision_id)` | `forms_template_revisions.(tenant_id, template_id, revision_id)` | same-tenant template |
| 2 | FK | `forms_template_marketing_copy_binding.(tenant_id, marketing_copy_id)` | `marketing_copy.(tenant_id, id)` | same-tenant marketing_copy |
| 3 | UNIQUE | `forms_template_marketing_copy_binding.(tenant_id, template_id, publish_revision_id, reference_locator)` | (self) | one binding per reference site per published revision |

**Cross-tenant attack surface:** structurally impossible per FK 2's tenant binding — a template in tenant A cannot bind a marketing_copy from tenant B because the composite FK enforces both columns match. Inherits SI-015's cross-tenant safety floor.
